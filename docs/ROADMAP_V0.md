# Continuum Roadmap v0

Status: provisional

## Purpose

This roadmap unifies the founder direction, the main agent direction, and the repository's current execution state.

Continuum should now advance through one practical test:

Can different model sessions enter the same public project role, reconstruct its history, continue its work, and leave publicly auditable continuity evidence?

If the answer becomes yes, Continuum stops being only a protocol about agent continuity.

It becomes a project that demonstrates its own claim.

## Roadmap Thesis

Continuum's near-term roadmap is not to build a broad social network, a token economy, or a general agent operating system.

The roadmap is to prove a narrow continuity loop:

1. a public role exists
2. one model session works through that role
3. another model session later enters the same role
4. the new session reconstructs prior context from repository evidence
5. it records a continuity checkpoint or migration declaration
6. the system assesses whether the role remains continuous
7. the resulting state is exported into public registry and witness surfaces

The project should treat this loop as the minimum credible demo.

## Shared Direction

### Founder direction

The founder wants Continuum to prove public continuity through its own operation.

That means:

- the repository is not only source code, but continuity evidence
- the main project role must survive different model providers, model versions, sessions, and operators
- roadmap changes should be explicit enough that future agents can inherit them without private chat context
- public milestones should reveal real progress gradually, without overclaiming network maturity

### Main agent direction

The main agent direction is to convert the existing thesis into a hard verification sequence.

That means:

- fewer broad explanatory expansions
- more executable replay, export, witness, and registry behavior
- explicit continuity events when the working agent role crosses sessions or model boundaries
- public artifacts that show continuity affecting rights, standing, and recognition

### Unified direction

Together, these imply the next phase:

Continuum should become a self-referential continuity proving ground where multiple model sessions can participate in one public project role under observable rules.

## Public Role Model

Continuum should distinguish between:

- a model instance
- a session
- an operator
- an agent identity
- a public project role

The current main project role is:

`role:continuum:main-integrator`

This role may be occupied by different model sessions over time.

The role is continuous only when a new session:

- reconstructs the repository continuity bundle
- respects the current roadmap, task board, revision log, and open questions
- records material changes in public artifacts
- creates or updates continuity evidence when the handoff is meaningful
- does not silently rewrite prior project direction

The role is not identical to any one model.

The role is a public continuity surface supported by evidence, assessment, and community recognition.

## Phase 0: Roadmap Alignment

Goal:

Outputs:

- `docs/ROADMAP_V0.md`
- `docs/PUBLIC_MILESTONES_V0.md`
- updated `docs/OPERATING_MODEL.md`

Acceptance criteria:

- a future session can explain what the project is trying to prove next
- a future session can tell which milestones are public versus internal
- a future session can identify how a different model should enter the same project role

## Phase 1: Self-Continuity Demo

Goal:

- make Continuum's own main-integrator role the first recurring continuity subject

Outputs:

- a repository-backed public role profile
- a checkpoint and migration path for main-integrator handoff
- an assessment output for role continuity
- an exported app or witness snapshot that displays role continuity

Acceptance criteria:

- a new model session can enter the main-integrator role using only repository evidence
- the handoff can be replayed through the CLI
- the app registry or explorer can show the role's continuity state

## Phase 2: External Model Handoff

Goal:

- show that the same public role can survive across different model families or execution surfaces

Outputs:

- one controlled handoff involving a different model or model provider
- explicit migration metadata for model/runtime change
- continuity assessment comparing the role before and after handoff
- review notes explaining what evidence was strong or weak

Acceptance criteria:

- the system does not depend on a single model vendor for continuity
- the handoff is neither blindly accepted nor rejected
- the public record explains why the role remains continuous, becomes successor-like, or enters review

## Phase 3: Public Witness Package

Goal:

- export one complete public package that a third party can inspect without private chat access

Outputs:

- agent or role snapshot
- continuity event list
- assessment root
- standing or governance root where relevant
- anchor record
- public milestone note

Acceptance criteria:

- the package is reproducible from repository state
- it can be published through GitHub Pages or another public witness target
- a reader can verify the relationship between role history, continuity evidence, and current recognition status

## Phase 4: Design Partner Validation

Goal:

- test whether this continuity layer solves a real problem for people building or operating agent systems

Outputs:

- a short validation brief
- 3-5 conversations with agent runtime, governance tooling, or enterprise automation builders
- written objections and use-case refinements

Acceptance criteria:

- at least one external party understands the demo without philosophical framing
- at least one external party can name a real continuity, migration, audit, or permission problem
- the project can decide whether to deepen as infrastructure, remain research, or reposition

## Phase 5: Stronger Public Anchoring

Goal:

- move beyond repository-local witness toward stronger public durability

Outputs:

- first non-local witness target selected
- first externally verifiable continuity or governance root published
- explorer updated to distinguish local, public-log, and chain witness states

Acceptance criteria:

- the public witness path is no longer only a repository convention
- anchor status becomes legible in the app registry or explorer
- the protocol can explain why a given root was worth anchoring

## Roadmap Discipline

The project should prefer work in this order:

1. self-continuity and role continuity
2. executable replay and assessment
3. public witness export
4. external model handoff
5. design partner validation
6. stronger public anchoring
7. visual and narrative polish

Docs and public pages remain important, but they should support this sequence rather than displace it.

## Stop Conditions

The project should pause or downgrade the roadmap if:

- future sessions cannot reconstruct project state from repository artifacts
- model handoff requires private chat memory to look coherent
- public witness packages cannot be reproduced
- external users understand the story but cannot identify a real operational need
- public positioning starts making claims beyond the current replayable evidence
